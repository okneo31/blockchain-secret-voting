# TTL 비밀투표 시스템 — 개발지시서 (Claude Code용)

> 작업 코드네임: **TTL Vote** (가칭, Grill Me 단계에서 확정)
> 체인 기반: TTL Coin (Chain ID 7777, Geth Clique PoA)
> 목표: 전국 규모(대통령 선거, 유권자 약 4,400만) 비밀투표 시스템

---

## 0. 작업 방식 — 반드시 먼저 읽을 것 (Matt Pocock 소프트웨어 펀더멘털)

**이 문서를 읽고 바로 코드를 짜지 마라.** 아래 순서를 반드시 지킨다.

1. **Grill Me (의도 캐묻기)** — 이 문서에서 모호하거나 결정이 필요한 부분을 먼저 질문으로 정리해서 나에게 물어본다. (질문 후보는 §10에 있음)
2. **Spec Agreement (명세 합의)** — 질문 답변을 받은 뒤, 우리가 만들 것의 명세를 글로 합의한다. 합의 전 구현 금지.
3. **Deep Module Design (깊은 모듈 설계)** — 모듈별 책임/인터페이스/숨기는 복잡성을 설계한다. 얕은 모듈(인터페이스가 구현만큼 복잡한 것) 금지.
4. **Implementation (구현)** — 합의된 설계 위에서만 코드를 쓴다.

**원칙**
- **Code is not cheap.** 코드를 쓰는 순간 평생 유지보수 비용이 생긴다. 꼭 필요한 것만.
- **유비쿼터스 언어 유지** — 도메인 용어(한글)를 코드에 일관되게 반영한다. 예: `Game::playCard()`가 아니라 `Game::naeda()`처럼. 본 프로젝트 용어집은 §4 참고.
- **깊은 모듈 지향** — 바깥에서 보는 인터페이스는 단순하게, 암호 복잡성은 모듈 안에 숨긴다.

## 1. 만들 것과 그 이유 (의도)

투표는 두 가지를 동시에 만족해야 하는데, 보통은 충돌한다.

- **투명성**: 누구나 "조작 없이 정확하게 집계됐다"를 검증할 수 있어야 한다.
- **비밀성**: "누가 누구를 찍었는지"는 그 누구도(운영자·정부 포함) 알 수 없어야 한다.

블록체인은 "모두가 같은 장부를 본다"는 점에서 투명성에 강하지만, 그대로 쓰면 표가 다 공개돼 비밀성이 깨진다. 그래서 **암호 기술로 비밀성을 덧씌운 채 투명성을 유지**하는 것이 이 시스템의 핵심이다.

### 4대 보장 (이 시스템이 무슨 일이 있어도 지켜야 하는 것)

| # | 보장 | 의미 | 담당 기술 |
|---|------|------|-----------|
| 1 | **1인 1표** | 유권자 1명은 정확히 1표만 | 무효화자(Nullifier) |
| 2 | **익명성** | 누가 무엇을 찍었는지 추적 불가 | 영지식증명 + 봉인표 |
| 3 | **결과 투명성** | 집계 결과를 누구나 검증 가능 | 동형암호 + 공개 검증 |
| 4 | **단독 조작 불가** | 한 기관이 혼자 결과를 열거나 바꿀 수 없음 | 임계 개표(9-of-15) |

이 네 가지는 **테스트로 강제**한다 (§9). 하나라도 깨지면 빌드 실패로 간주한다.

## 2. 핵심 요구사항

### 기능 요구사항
- 유권자 등록 → 투표권 발급 → 봉인투표 → 집계 → 임계 개표 → 공개 검증의 전체 생애주기 지원
- 이중투표 시도는 **거부**하되, 시도자가 누구인지는 **알 수 없어야** 함
- 투표 종료 시각 전에는 누구도(개표위원 포함) 중간 결과를 열 수 없어야 함
- 모든 단계가 **사후에 독립 검증 가능** (검증 도구 별도 제공)

### 비기능 요구사항
- **규모**: 유권자 4,400만, 투표 피크 시간대 집중 트래픽 가정
- **검증 시간**: 전체 집계 검증이 일반 PC에서 현실적인 시간 안에 끝나야 함
- **장애 내성**: 개표위원 15명 중 6명이 키를 잃거나 비협조해도 개표 가능(9명이면 됨)
- **투명성**: 검증 코드와 회로(circuit)는 전면 공개(오픈소스)

## 4. 유비쿼터스 언어 (도메인 용어집) — 코드 전체에서 통일

암호 표준 용어(국제 통용)는 영어를 유지하고, 업무 도메인 동작은 한글(로마자)을 따른다. 혼용 금지.

| 한글 도메인 용어 | 의미 | 코드 식별자(제안) |
|------------------|------|-------------------|
| 유권자명부 | 자격 있는 유권자들의 머클트리 | `VoterRegistry` |
| 신원약속값 | 유권자 비밀에서 파생한 공개 약속값 | `identityCommitment` |
| 익명집합 | 명부에 든 전체 유권자 집합(숨을 군중) | `anonymitySet` |
| 투표권 | 신원과 분리된 투표 자격 토큰 | `credential` |
| 봉인표 | 암호화된 표(내용 안 보임) | `sealedBallot` |
| 무효화자 | 1인 1표 보장용 유일 해시 | `nullifier` |
| 게시판 | 모든 기록이 올라가는 공개 장부(=체인) | `bulletinBoard` |
| 집계 | 봉인표들을 암호 상태로 합산 | `tally` |
| 개표 | 합산된 암호값을 복호화하는 행위 | `decrypt` |
| 개표위원 | 복호화 키 조각을 가진 신뢰 기관 | `trustee` |
| 임계개표 | t명 이상 모여야 가능한 복호화 | `thresholdDecrypt` |
| 봉인투표(동작) | 표를 봉인해 게시판에 올리는 행위 | `bonginTupyo()` |

> 예) 투표 동작은 `vote()`가 아니라 `bonginTupyo()`로 명명한다. 도메인 언어가 코드에 그대로 살아야 한다. (최종 명명은 Grill Me에서 합의)

## 5. 모듈 설계 (Deep Modules)

각 모듈은 **단순한 인터페이스 + 안에 숨긴 복잡성**을 원칙으로 한다. 아래는 책임 분해 초안이며, 인터페이스 시그니처는 설계 단계에서 확정한다.

### 5.1 유권자명부 모듈 (`VoterRegistry`)
- **책임**: 자격 유권자의 신원약속값을 머클트리로 관리하고 루트를 게시판에 올린다.
- **인터페이스(개념)**: `등록(신원약속값)`, `루트가져오기()`, `멤버십증거생성(약속값)`
- **숨기는 복잡성**: 머클트리 구현, Poseidon 해시, 증분 갱신, 루트 버저닝
- **불변식**: 한 번 등록된 약속값은 임의 삭제 불가, 루트 변경은 항상 게시판에 기록

### 5.2 투표권 발급 모듈 (`CredentialIssuer`)
- **책임**: 본인 인증을 거친 유권자에게 **신원과 연결되지 않는** 투표권을 발급한다.
- **인터페이스(개념)**: `발급요청(본인인증)`, `발급(블라인드서명)`
- **숨기는 복잡성**: 블라인드 서명, 신원-투표권 분리, 발급 중복 차단
- **불변식**: 발급자조차 "이 투표권이 누구 것인지" 알 수 없다

### 5.3 봉인투표 모듈 (`BallotCasting`)
- **책임**: 투표자가 (1) 자격 증명 (2) 봉인표 생성 (3) 무효화자 제출을 한 번에 수행한다.
- **인터페이스(개념)**: `bonginTupyo(선택, 멤버십증거) → (봉인표, 무효화자, 영지식증명)`
- **숨기는 복잡성**:
  - 영지식증명 생성: "나는 명부에 있다 + 무효화자를 올바르게 만들었다 + 봉인표가 유효한 선택지다"
  - 봉인표 암호화(동형암호 공개키 사용)
  - 선택지 유효성 증명(찬성/반대 등 허용 범위 안인지 = 범위 증명)
- **불변식**: 봉인표·무효화자·증명 어디에도 신원 정보가 없다

### 5.4 게시판/체인 모듈 (`BulletinBoard`)
- **책임**: 명부 루트, 무효화자 집합, 봉인표(또는 그 약속값), 집계 앵커, 개표 증명을 불변으로 기록한다.
- **인터페이스(개념)**: `무효화자등록(무효화자)`, `봉인표기록(...)`, `이미사용됨?(무효화자)`
- **숨기는 복잡성**: 스마트컨트랙트 호출, 가스/배치, 온체인-오프체인 데이터 분리(§6 참고)
- **불변식**: 같은 무효화자 두 번 등록 시도는 **거부**

### 5.5 집계 모듈 (`Tally`)
- **책임**: 봉인표들을 **복호화하지 않고** 암호 상태 그대로 합산한다.
- **인터페이스(개념)**: `집계(봉인표목록) → 합산봉인값`
- **숨기는 복잡성**: 동형 덧셈, 누락/위조 봉인표 필터링, 합산 과정의 검증 증거 생성
- **불변식**: 개별 표는 끝까지 열리지 않는다

### 5.6 임계 개표 모듈 (`ThresholdDecryption`)
- **책임**: 분산 생성된 키 조각을 9명 이상이 모아 **최종 합산값만** 복호화한다.
- **인터페이스(개념)**: `부분복호화(개표위원, 합산봉인값)`, `결합(부분복호화목록) → 최종결과`
- **숨기는 복잡성**: DKG, 임계 ElGamal, 부분 복호화 정당성 증명, 비협조 위원 처리
- **불변식**: 9명 미만으로는 절대 결과가 열리지 않는다. 개별 표는 임계 개표로도 못 연다(합산값만 대상)

### 5.7 검증 모듈 (`Verifier`) — 별도 공개 도구
- **책임**: 외부인이 전 과정을 독립 검증한다.
- **검증 항목**: 명부 루트 정합성 · 모든 영지식증명 · 무효화자 유일성 · 동형 합산 정확성 · 개표 복호화 증명
- **불변식**: 검증에 **운영자의 협조나 비밀이 전혀 필요 없다**

## 6. 온체인 / 오프체인 분리 (규모 현실)

> 솔직한 제약: 유권자 4,400만 규모에서 **모든 봉인표 원본을 PoA 체인에 직접 올리는 것은 비현실적**(저장/처리 비용). 체인은 "무결성 앵커" 역할로 쓴다.

| 데이터 | 위치 | 이유 |
|--------|------|------|
| 명부 머클 루트 | 온체인 | 위변조 즉시 탐지 |
| 무효화자 집합(또는 누적 약속) | 온체인 | 이중투표 차단의 진실 원천 |
| 봉인표 원본 | 오프체인(가용성 계층) + 온체인 해시/루트 | 비용 ↓, 무결성은 앵커로 보장 |
| 집계 결과 + 개표 증명 | 온체인 | 최종 결과의 공개 검증 |

이 분리는 **반드시 Spec Agreement에서 합의**하고, 오프체인 데이터의 가용성·복제·공증 방식을 명확히 한다.

## 7. 보안 요구사항 / 위협 모델

방어해야 할 시나리오:

1. **이중투표** → 무효화자로 차단 (시도자 익명 유지)
2. **표 매수/강요(매표)** → 영수증 없는 설계: 투표자가 "이렇게 찍었다"를 제3자에게 증명 불가하게 (필요 시 MACI식 키 교체 검토 — Grill Me 항목)
3. **운영자의 표 열람** → 운영자는 복호화 키가 없음. 임계 개표 위원만, 그것도 합산값만.
4. **개표위원 담합** → 9-of-15. 9곳 동시 포섭은 비현실적.
5. **결과 조작** → 모든 단계 공개 증명. 조작 시 검증 단계에서 즉시 발각.
6. **중간 결과 유출로 추세 편승** → 종료 전 개표 봉쇄(시간잠금 또는 위원 비공개 합의).
7. **명부 위조** → 머클 루트 온체인 공개 + 등록 절차 감사.

**금지**: 어떤 단계에서도 "신원 ↔ 표"를 잇는 데이터를 저장·로깅하지 않는다. 디버그 로그에도 금지.

## 8. 단계별 구현 로드맵

> 각 단계는 끝에 **검증 가능한 산출물**을 남긴다. 단계 건너뛰기 금지.

- **0단계 — 합의**: Grill Me → Spec Agreement → 모듈 인터페이스 확정
- **1단계 — 명부 + 멤버십 증명**: 머클트리, 신원약속값, "명부에 있다" 영지식증명 (이중투표/봉인 없이 익명 멤버십만)
- **2단계 — 무효화자**: 1인 1표 강제 + 익명 유지 테스트
- **3단계 — 봉인표 + 동형 합산**: 표 암호화, 암호 상태 합산, 선택지 유효성 증명
- **4단계 — 임계 개표**: DKG, 9-of-15 복호화, 부분 복호화 증명
- **5단계 — 게시판/체인 통합**: 온체인 앵커, 오프체인 가용성
- **6단계 — 검증 도구**: 외부 독립 검증 CLI/웹
- **7단계 — 매표 방어 강화**: 영수증 없는 설계 마감
- **8단계 — 규모 시험**: 대량 부하·검증 시간 측정

## 9. 테스트 전략 — 4대 보장을 코드로 강제

각 보장마다 "깨지면 빌드 실패"인 테스트를 둔다.

- **1인 1표**: 같은 비밀로 두 번 투표 → 두 번째는 거부. 무효화자 충돌 검출 테스트.
- **익명성**: 봉인표·무효화자·증명 어디에도 신원 복원이 불가함을 단언하는 테스트(연결 불가성).
- **결과 투명성**: 임의로 만든 표 묶음의 합산 결과를, 운영자 비밀 없이 외부 검증 도구만으로 재검증.
- **단독 조작 불가**: 개표위원 8명만으로 복호화 시도 → 실패. 1명이 단독 복호화 시도 → 실패. 9명 → 성공.
- **매표 저항**: 투표자가 자기 표를 제3자에게 증명하려는 경로가 막혀 있는지 점검.

추가: 회로 제약(circuit constraints) 단위 테스트, 동형 합산 결합법칙 속성 테스트(property-based test).

## 10. Claude Code 첫 지시 — 여기서 시작 (Grill Me)

**바로 코드 짜지 말고, 아래 항목들을 먼저 나에게 질문해서 답을 받아라.**

1. 코드네임/네이밍: TTL Vote 확정? 도메인 동작 식별자는 로마자 한글(`bonginTupyo`)로 통일할까, 영어로 갈까?
2. 익명 멤버십을 **Semaphore 차용**으로 갈까, **자체 회로**로 갈까? (유지보수·감사 vs 통제권)
3. 동형암호: **지수형 ElGamal** vs **Paillier**? (집계 규모·복호화 난이도 트레이드오프)
4. 매표 방어를 어디까지? 영수증 없는 설계만으로 충분한가, **MACI식 키 교체(조정자 도입)**까지 갈까?
5. 온체인/오프체인 분리 정책(§6) 확정 — 봉인표 원본 가용성 계층은 무엇으로?
6. 개표위원 15곳을 누구로 상정? (테스트넷에서는 더미여도, 키 의식(ceremony) 절차 설계 필요)
7. 종료 전 개표 봉쇄: **시간잠금 암호** vs **위원 비공개 합의** 중 무엇?
8. 1차 목표 산출물: 전체 시스템 vs §8의 1~2단계 PoC 먼저?

이 8개에 대한 답을 받은 뒤 **Spec Agreement 문서**를 작성해 다시 합의한다. 그 다음 모듈 설계, 그 다음 구현.

### 부록 A. 한 표가 처리되는 흐름 (요약 시퀀스)

1. 유권자가 본인 인증 → **투표권 발급**(신원과 분리)
2. 투표자가 비밀에서 **신원약속값**과 **무효화자** 파생
3. 클라이언트가 **영지식증명** 생성: "명부에 있음 + 무효화자 올바름 + 봉인표 유효"
4. **봉인표 + 무효화자 + 증명**을 게시판에 제출 (신원 없음)
5. 게시판이 무효화자 중복 확인 → 통과 시 기록
6. 종료 후 모든 **봉인표를 암호 상태로 합산**
7. 개표위원 **9명 이상**이 모여 합산값만 **임계 개표**
8. 결과 + 증명 공개 → 누구나 **독립 검증**
